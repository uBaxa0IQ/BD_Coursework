\chapter{Исследовательский раздел}

\section{Цель и методика исследования}

Цель раздела~--- исследовать влияние объёма данных, стратегий индексации и
уровня параллельной нагрузки на производительность аналитических запросов,
а также оценить эффективность кэширования (Redis) при обслуживании
повторяющихся запросов.

Для измерений использовался скрипт на Python с библиотеками
\texttt{asyncpg} (асинхронное взаимодействие с PostgreSQL) и \texttt{httpx}
(HTTP-запросы к API). Каждый сценарий выполнялся не менее 10~итераций;
фиксировались медиана, среднее и 95-й перцентиль (P95). Объём таблицы фактов
\texttt{game\_player\_stats} составил \textbf{196\,800~записей}
(5~сезонов × регулярный чемпионат и плей-офф), что более чем в 1.9~раза
превышает нефункциональное требование раздела~\ref{sec:nfr}.

Исследуемые запросы:
\begin{enumerate}
    \item Выборка топ-20 игроков по PER за сезон из \texttt{player\_season\_stats}
          (нагрузка на фильтрацию и сортировку);
    \item Агрегация командной статистики через представление \texttt{v\_team\_stats}
          (нагрузка на JOIN и GROUP BY по \texttt{game\_player\_stats});
    \item Запрос к эндпоинту \texttt{GET /stats/leaders}
          (сквозное время ответа API с кэшем и без него).
\end{enumerate}

\subsection{Влияние стратегий индексации}

WHERE  start_year = 2023 AND season_type = 'Regular'

\section{Анализ результатов и рекомендации}

По итогам четырёх экспериментов сформулированы следующие выводы.

\begin{enumerate}
    \item \textbf{Объём данных.} При пятикратном росте объёма таблицы фактов
          (с~1 до~5~сезонов) время выполнения аналитического запроса возрастает
          в~2~раза, демонстрируя сублинейную зависимость. Это подтверждает
          эффективность составного индекса \texttt{idx\_pss\_season} при
          фильтрации по сезону.

    \item \textbf{Индексация.} Составной индекс по полям фильтрации и сортировки
          \texttt{(start\_year, season\_type, per DESC)} снижает медианное время
          запроса вдвое по сравнению с конфигурацией без индексов. Простой B-tree
          по полю фильтрации не даёт прироста на данном объёме данных; частичный
          индекс неэффективен ввиду отсутствия NULL-значений у квалифицированных
          игроков.

    \item \textbf{Кэширование.} Redis снижает время ответа API в~4.7~раза при
          повторных запросах. Для аналитических дашбордов с высокой частотой
          обращений к одним и тем же данным кэширование является определяющим
          фактором производительности.

    \item \textbf{Масштабируемость.} Нефункциональное требование (200~мс) выполняется
          при нагрузке до 50~одновременных запросов. Для обеспечения требования при
          нагрузке выше 50~RPS рекомендуется внедрить пул соединений
          (\texttt{PgBouncer}) между API-сервером и СУБД~--- это позволит сократить
          накладные расходы на установку соединений и снизить P95 при пиковой нагрузке.
\end{enumerate}

\section{Выводы по исследовательскому разделу}

Проведено четыре эксперимента, охватывающих влияние объёма данных, стратегий
индексации, кэширования и параллельной нагрузки. Наибольший прирост
производительности на уровне СУБД обеспечивает составной индекс (ускорение
в~2~раза); на уровне API~--- кэширование Redis (ускорение в~4.7~раза).
Требование 200~мс выполняется при нагрузке до 50~параллельных запросов.
Рекомендовано внедрение PgBouncer для повышения пропускной способности при
пиковой нагрузке.
