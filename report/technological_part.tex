\section{Выбор и обоснование средств реализации}

Разрабатываемая система является дата-интенсивным приложением, в котором основная вычислительная нагрузка ложится на слой хранения данных. Выбор технологического стека обусловлен необходимостью поддержания строгой транзакционной целостности (ACID), выполнения сложных аналитических расчётов и обеспечения высокой скорости чтения с помощью кэширования.

\textbf{Слой базы данных.} В качестве основной СУБД выбрана PostgreSQL 15. В отличие от рассматриваемых альтернатив (MySQL 8, MS SQL Server), PostgreSQL является свободно распространяемой системой с мощным встроенным языком PL/pgSQL. Данный язык критически важен для реализации алгоритмов расчёта спортивных метрик непосредственно на стороне хранилища. Кроме того, PostgreSQL обладает нативной поддержкой конструкции \texttt{UPSERT} и расширенными возможностями проверки ограничений целостности (\texttt{CHECK}).

\textbf{Слой кэширования.} Для снижения нагрузки на дисковую подсистему при повторных аналитических запросах выбрана in-memory СУБД Redis 7. В отличие от Memcached, Redis поддерживает сложные структуры данных и позволяет выполнять эффективную инвалидацию кэша по префиксу ключа (pattern matching).

\textbf{Серверный и клиентский слои.} Программный интерфейс (REST API) реализован на языке Python с применением асинхронного фреймворка FastAPI и драйвера \texttt{asyncpg}, обеспечивающего высокопроизводительное неблокирующее взаимодействие с базой данных. Клиентская часть разработана в виде Single Page Application (SPA) на базе React/TypeScript, чья задача сводится к маршрутизации REST-запросов и визуализации полученной статистики.

\subsection{Загрузка тестовых данных}

Первоначальное наполнение базы данных выполнялось программным ETL-скриптом
\texttt{load\_data.py} на языке Python. Скрипт извлекает архивную статистику NBA
за пять регулярных сезонов (2019/20--2023/24) из официального портала
\texttt{NBA Stats}~\cite{nba-stats}. Доступ к протоколам матчей (Box Score)
и справочникам лиги реализован через библиотеку \texttt{nba\_api}~\cite{nba-api},
формирующую HTTP-запросы к сервису \texttt{stats.nba.com} и возвращающую
структурированные ответы (JSON).

Полученные данные нормализуются в скрипте и загружаются в реляционные таблицы
PostgreSQL пакетами (batch insert) с соблюдением порядка внешних ключей. Данный подход гарантирует происхождение данных (data provenance) из официального источника и сохраняет естественное распределение показателей реальных игроков. Естественное распределение критически важно для корректной верификации относительных метрик (таких как BPM и PER).

\section{Реализация логики предметной области в СУБД}

\subsection{Хранимые функции расчёта метрик}

Все математические алгоритмы расчёта метрик перенесены на слой СУБД и реализованы в виде хранимых функций на языке PL/pgSQL. Исходный код всех функций вынесён в Приложение~Б. 

В листинге~\ref{lst:ts} приведён фрагмент функции \texttt{calculate\_ts}, вычисляющей истинный процент бросков (TS\%) с программной защитой от деления на ноль.

\begin{lstlisting}[language=SQL, label=lst:ts, caption=Реализация функции \texttt{calculate\_ts} (фрагмент)]
CREATE OR REPLACE FUNCTION calculate_ts(
    p_player_id INT, p_season_id INT, p_team_id INT DEFAULT NULL
) RETURNS DECIMAL(5,4) LANGUAGE plpgsql STABLE AS $$
DECLARE
    v_pts NUMERIC; v_fga NUMERIC; v_fta NUMERIC; v_denom NUMERIC;
BEGIN
    IF v_fga = 0 AND v_fta = 0 THEN RETURN NULL; END IF;
    
    v_denom := 2.0 * (v_fga + 0.44 * v_fta);
    IF v_denom = 0 THEN RETURN NULL; END IF;
    
    RETURN ROUND(v_pts::DECIMAL / v_denom, 4);
END; $$;
\end{lstlisting}

Все расчётные функции помечены квалификатором \texttt{STABLE}, что позволяет оптимизатору запросов кэшировать результаты вызовов в рамках одного оператора. Агрегация сезонной статистики выполняется хранимой процедурой \texttt{update\_season\_stats} с использованием атомарной конструкции \texttt{INSERT ... ON CONFLICT DO UPDATE}.

\subsection{Триггеры логической целостности}

Для защиты данных от аномалий реализован триггер \texttt{trg\_check\_team\_score}. Он проверяет фундаментальное правило предметной области: сумма очков всех игроков команды в конкретном матче должна быть равна итоговому командному счёту.

\begin{lstlisting}[language=SQL, label=lst:trigger, caption=Триггер проверки корректности очков (фрагмент)]
CREATE OR REPLACE FUNCTION trg_validate_team_score_row()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
    v_sum INT; v_expected INT;
BEGIN
    IF v_expected IS NOT NULL AND v_sum <> v_expected THEN
        RAISE EXCEPTION
            'Player points sum (%) does not match team score (%)',
            v_sum, v_expected;
    END IF;
    RETURN COALESCE(NEW, OLD);
END; $$;
\end{lstlisting}

\section{Реализация ролевой модели доступа}

Разграничение прав реализовано с использованием ролей базы данных (\texttt{db\_admin}, \texttt{analyst}, \texttt{reader}). \section{Реализация программного обеспечения и интерфейса}

\subsection{Слой маршрутизации и взаимодействия с БД (Backend)}

Серверная часть приложения выполняет роль прослойки доступа к данным
(Data Access Layer). Бэкенд реализован на фреймворке FastAPI в соответствии
со спецификацией REST API конструкторского раздела (подраздел~\ref{subsec:api-spec}).
Для взаимодействия с PostgreSQL применяется асинхронный драйвер \texttt{asyncpg}
в связке с ORM SQLAlchemy~2. Движки подключений настроены с пулом соединений
(Connection Pooling, \texttt{pool\_size=10}, \texttt{max\_overflow=20}), что
позволяет СУБД обслуживать конкурентные запросы аналитиков без постоянного
открытия новых TCP-сессий.

Жизненный цикл сессии базы данных управляется через механизм внедрения
зависимостей (Dependency Injection): для каждого HTTP-запроса создаётся
экземпляр \texttt{AsyncSession}, привязанный к роли пользователя СУБД
(\texttt{analyst}, \texttt{reader}, \texttt{db\_admin}), и гарантированно
закрывается по завершении обработки маршрута. Для снижения нагрузки
на дисковую подсистему реализован паттерн Cache-Aside на базе Redis
(класс \texttt{CacheManager}). В листинге~\ref{lst:fastapi} показан фрагмент
маршрута \texttt{GET /players/\{player\_id\}/stats}; полный исходный код
вынесён в приложение~В.

\begin{lstlisting}[language=Python, label=lst:fastapi,
  caption=Реализация взаимодействия с БД и кэшем (фрагмент)]
@router.get("/{player_id}/stats")
async def get_player_stats(
    player_id: int,
    season_id: int | None = None,
    cache: CacheManager = Depends(get_cache),
    db: AsyncSession = Depends(get_db_analyst),
):
    cache_key = f"player_stats:{player_id}:{season_id or 'all'}"
    cached = await cache.get(cache_key)
    if cached:
        return cached

    query = (
        select(PlayerSeasonStats.season_id, PlayerSeasonStats.ts_pct, ...)
        .where(PlayerSeasonStats.player_id == player_id)
    )
    if season_id:
        query = query.where(PlayerSeasonStats.season_id == season_id)

    result = await db.execute(query)
    data = [dict(row) for row in result.mappings().all()]
    await cache.set(cache_key, data, ttl=300)
    return data
\end{lstlisting}

\subsection{Пользовательский интерфейс (Frontend)}

Клиентское веб-приложение реализовано на React (TypeScript) и предоставляет
графический интерфейс для анализа данных, извлекаемых из СУБД через REST API.
Приложение не дублирует бизнес-логику расчёта метрик~--- все агрегаты
формируются на стороне PostgreSQL (функции, представления, витрины).
Основные разделы интерфейса:
\begin{enumerate}
    \item \textbf{Dashboard}~--- сводная статистика лиги, последние матчи
          и ключевые показатели сезона (данные из представлений и агрегатов СУБД);
    \item \textbf{Players / Teams}~--- справочники с серверным поиском
          (маршрут \texttt{/league/search}, индекс \texttt{idx\_players\_name})
          и фильтрацией по сезону и команде;
    \item \textbf{Leaderboard / Standings}~--- рейтинги лидеров и турнирная
          таблица по метрикам PTS, AST, REB, PER, TS\%, eFG\%, BPM;

\section{Тестирование и верификация аналитических данных}

Тестирование логики СУБД проводилось методом чёрного ящика. Для проверки точности математических агрегаций выполнена ручная верификация на реальных данных из загруженной базы.

В качестве тестового объекта был выбран профиль игрока Nikola Joki\'{c} (\texttt{player\_id = 2327}) за регулярный сезон 2023-24 (\texttt{season\_id = 5}). Прямой SQL-запрос суммы сырых фактов из таблицы \texttt{game\_player\_stats} вернул следующие значения:
\begin{itemize}
    \item очки (PTS): 2085;
    \item броски с игры (FGA): 1411;
    \item штрафные попытки (FTA): 438.
\end{itemize}

Ручной расчёт истинного процента бросков (TS\%) по формуле~(\ref{eq:ts}):
\[
\mathrm{TS\%} = \frac{2085}{2 \cdot (1411 + 0{,}44 \cdot 438)}
  = \frac{2085}{2 \cdot 1603{,}72}
  = \frac{2085}{3207{,}44} \approx 0{,}6501.
\]

Для верификации в СУБД был выполнен вызов разработанной хранимой функции:
\begin{lstlisting}[language=SQL]
SELECT calculate_ts(2327, 5);
\end{lstlisting}

Результат, возвращённый базой данных, абсолютно совпал с ручным аналитическим расчётом. Аналогичная верификация проведена для метрик eFG\% и USG\%; расхождений не выявлено. Тестирование граничных случаев (отсутствие попыток бросков, нулевое время на площадке) подтвердило корректную обработку исключений и возврат значения \texttt{NULL} без прерывания транзакций (таблица~\ref{tab:test-cases}).

\section*{Вывод}

В технологическом разделе обоснован программный стек: PostgreSQL, Redis, FastAPI и React. Реализована физическая реляционная схема базы данных, включающая ссылочные ограничения и структуры B-деревьев. Разработанный ETL-скрипт обеспечил загрузку более 150~тысяч реальных фактов статистики из открытых источников. Вся расчётная логика предметной области успешно перенесена на уровень хранилища в виде функций PL/pgSQL и триггеров целостности. Реализована ролевая модель доступа, серверное REST API с пулом соединений
и кэшированием Cache-Aside, а также клиентский интерфейс на React.
Проведённая верификация с ручным расчётом метрик на живых данных подтвердила полную математическую корректность функционирования базы данных.
