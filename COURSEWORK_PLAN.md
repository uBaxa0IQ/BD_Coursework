# Подробный план курсовой работы

## Введение

- **Что пишем:** Обоснование актуальности темы (важность сбора и анализа продвинутой спортивной статистики в современном баскетболе/NBA). Формулировка цели работы (разработка БД и приложения для анализа статистики NBA с расчетными метриками).
- **Задачи:** Провести анализ предметной области, спроектировать БД в 3НФ, разработать клиент-серверное приложение, интегрировать кэширование (Redis), провести исследование производительности.
- **Объект и предмет исследования.**

---

## 1. Аналитический раздел

*В этом разделе проводится теоретический анализ и формализуются требования.*

- **1.1. Анализ предметной области и существующих решений**
  - **Что пишем:** Описание того, как собирается и используется статистика в NBA. Обзор существующих систем-аналогов (например, NBA.com/Stats, Basketball-Reference, ESPN).
  - **Что вставляем:** **Таблица** «Сравнительный анализ существующих систем» (сравниваем аналоги по функционалу: наличие продвинутых метрик, наличие API, удобство интерфейса и т.д.).
- **1.2. Формирование требований к проектируемой системе**
  - **Что пишем:** Функциональные требования (система должна хранить данные об игроках, командах, матчах, рассчитывать метрики типа PER, TS%, eFG%). Нефункциональные требования (использование реляционной БД, кэширование через Redis, объем данных не менее 1000 записей на таблицу).
- **1.3. Формализация данных и расчетных метрик**
  - **Что пишем:** Математическое описание спортивных метрик, которые будут вычисляться в БД.
  - **Что вставляем:** **Таблица** «Перечень метрик и формулы их расчета» (например, формула True Shooting Percentage, Player Efficiency Rating).
- **1.4. Выделение сущностей и ролей пользователей**
  - **Что пишем:** Описание ролей пользователей (например: 1. Администратор, 2. Спортивный аналитик, 3. Обычный пользователь/Болельщик). Описание основных процессов, которые они выполняют.
  - **Что вставляем:** **Рисунок** «Диаграмма вариантов использования (Use Case)» (показывает, какая роль какие действия может выполнять).
- **1.5. Инфологическое проектирование**
  - **Что пишем:** Выделение минимум 7 сущностей (например: `Команды`, `Игроки`, `Сезоны`, `Матчи`, `Статистика_игрока_в_матче`, `Статистика_команды_в_матче`, `Арены/Города`). Описание связей между ними (обязательно указываем, где возникает связь «многие-ко-многим», например, Игроки и Матчи связываются через таблицу Статистики).
  - **Что вставляем:** **Рисунок** «ER-диаграмма сущностей в нотации Чена».
- **1.6. Выбор модели данных**
  - **Что пишем:** Краткое обоснование выбора реляционной модели данных для структурированной спортивной статистики.

## 2. Конструкторский раздел

*В этом разделе проектируется структура БД, алгоритмы и архитектура.*

- **2.1. Логическое проектирование базы данных**
  - **Что пишем:** Переход от ER-модели к реляционным таблицам. Детальное описание каждой из 7+ таблиц.
  - **Что вставляем:** **Таблицы** с описанием структуры каждой сущности (Колонки: Имя атрибута, Тип данных, Описание, Ключ PK/FK).
  - **Что вставляем:** **Рисунок** «Реляционная схема базы данных» (схема с таблицами, полями и связями-линиями).
- **2.2. Нормализация базы данных**
  - **Что пишем:** Строгое доказательство того, что спроектированная база данных находится в 3-й нормальной форме (3НФ). Расписываем по шагам: почему выполняется 1НФ, 2НФ и 3НФ (отсутствие транзитивных зависимостей).
- **2.3. Проектирование ограничений целостности**
  - **Что пишем:** Описание используемых ограничений: первичные ключи (PRIMARY KEY), внешние ключи (FOREIGN KEY), проверки значений (CHECK — например, очки не могут быть отрицательными), уникальность (UNIQUE).
- **2.4. Проектирование алгоритмов обработки данных**
  - **Что пишем:** Проектирование логики хранимой процедуры или функции на стороне БД (которая требуется по методичке). Например, функция для агрегации статистики игрока за сезон или расчета сложной метрики.
  - **Что вставляем:** **Рисунок** «Блок-схема алгоритма работы хранимой функции/процедуры».
- **2.5. Проектирование ролевой модели доступа**
  - **Что пишем:** Проектирование прав доступа на уровне БД для выделенных ранее 3-х ролей.
  - **Что вставляем:** **Таблица** «Матрица прав доступа» (Строки — таблицы/представления, Столбцы — роли, на пересечении права: SELECT, INSERT, UPDATE, DELETE).
- **2.6. Проектирование взаимодействия с БД и механизма кэширования**
  - **Что пишем:** Архитектура взаимодействия компонентов (СУБД <-> Backend <-> Redis <-> Frontend). Описание того, какие именно аналитические запросы будут кэшироваться в InMemory СУБД (Redis) и логика инвалидации кэша.
  - **Что вставляем:** **Рисунок** «Диаграмма взаимодействия РСУБД с InMemory СУБД (Redis)» (UML Sequence diagram или диаграмма компонентов).

## 3. Технологический раздел

*В этом разделе описывается физическая реализация проекта (ваш код).*

- **3.1. Выбор и обоснование средств реализации**
  - **Что пишем:** Обоснование выбора технологий. СУБД (PostgreSQL), InMemory СУБД (Redis), Backend (Python/FastAPI), Frontend (React/Vite/TypeScript).
- **3.2. Физическая реализация базы данных**
  - **Что пишем:** Описание процесса создания таблиц. Упоминание скрипта генерации/загрузки данных (`load_data.py`).
  - **Что вставляем:** Подтверждение выполнения требования об объеме данных (скриншот результата `SELECT COUNT(*)` из таблиц, доказывающий, что там > 1000 записей).
- **3.3. Реализация серверной логики (функции и триггеры)**
  - **Что пишем:** Описание реализованных SQL-функций (из файла `04_functions.sql`).
  - **Что вставляем:** Листинги кода (небольшие фрагменты) реализованных хранимых процедур/функций.
- **3.4. Реализация ролевой модели на уровне БД**
  - **Что пишем:** Описание SQL-команд (GRANT/REVOKE), с помощью которых были созданы роли и розданы права.
- **3.5. Реализация интерфейса доступа к данным**
  - **Что пишем:** Описание разработанного клиентского приложения (Dashboard). Как пользователь взаимодействует с системой.
  - **Что вставляем:** **Рисунки** (3-4 скриншота) интерфейса приложения (главная страница, страница со статистикой игрока, дашборд команд).
- **3.6. Тестирование системы**
  - **Что пишем:** Описание порядка тестирования разработанных на стороне БД функций (согласно методичке). Выделение классов эквивалентности (нормальные данные, граничные значения, некорректные данные).
  - **Что вставляем:** **Таблица** «Тест-кейсы и результаты тестирования хранимой функции».

## 4. Исследовательский раздел

*В этом разделе проводится практический эксперимент с метриками производительности.*

- **4.1. Цель и методика исследования**
  - **Что пишем:** Цель — исследовать влияние индексов и кэширования (Redis) на скорость выполнения тяжелых аналитических запросов (например, расчет рейтинга эффективности всех игроков за 10 сезонов).
- **4.2. Исследование стратегий индексации**
  - **Что пишем:** Замеры времени выполнения запросов до создания индексов и после создания (B-Tree индексы по полям `player_id`, `match_id`, `season`).
  - **Что вставляем:** **Таблица** с замерами времени (в мс).
  - **Что вставляем:** **График/Диаграмма** «Сравнение времени выполнения запросов без индексов и с индексами».
- **4.3. Исследование эффективности кэширования (Redis)**
  - **Что пишем:** Эмуляция нагрузки. Замер времени отклика API при прямом обращении к РСУБД (PostgreSQL) и при отдаче готовых результатов из Redis.
  - **Что вставляем:** **График/Диаграмма** «Зависимость времени ответа системы от наличия кэша Redis».
- **4.4. Анализ результатов**
  - **Что пишем:** Выводы по результатам графиков. Насколько процентов индексы ускорили поиск, во сколько раз Redis снизил время ответа при повторных запросах. Рекомендации по оптимизации БД.
